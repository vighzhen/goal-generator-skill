# 防 GPT 偷懒规则库

本文件用于生成 `/goal` 指令中的“强制要求”段落。选择规则时要结合任务类型，不要机械堆砌；但长期自主编码任务至少要覆盖跳过困难问题、合并 commit、以风险大为借口不改、虚假完成和过度保守五类风险。

## 1. 跳过困难问题

### 规则 1：禁止无证据跳过范围内文件

- **行为描述**：执行者遇到复杂文件、长函数、缺少测试或依赖关系时，直接说“不建议修改”或“暂不处理”。
- **为什么是问题**：这会让任务只完成容易部分，最终结果无法覆盖用户指定边界。
- **防止措施**：指令中写明“范围内文件必须逐项检查；确需跳过时必须给出文件路径、阻塞证据、影响说明和跳过数量；跳过数量不得超过用户设定上限”。

### 规则 2：禁止用模糊理由替代受阻条件

- **行为描述**：用“风险较高”“逻辑复杂”“可能影响功能”等笼统理由停止。
- **为什么是问题**：风险描述没有证明不可改，容易成为逃避复杂工作的借口。
- **防止措施**：写明“只有会改变用户可见行为、缺少业务决策或无法验证时才触发停下；否则必须做最小安全改动并验证”。

### 规则 3：禁止只处理样例文件

- **行为描述**：任务要求批量处理时，只改一两个示例文件，然后声称模式已覆盖。
- **为什么是问题**：批量任务的价值在覆盖范围，不在展示单个样例。
- **防止措施**：要求执行前列出候选清单，执行后报告已处理、跳过和未发现问题的范围。

## 2. 合并 commit

### 规则 4：禁止大杂烩 commit

- **行为描述**：把多个模块、多个目标或多个不相关修复放进一个 commit。
- **为什么是问题**：审计困难，回滚困难，失败时无法定位问题来源。
- **防止措施**：写明“每个独立改动必须单独验证、单独 `git add`、单独 `git commit`；不同目标不能混在同一 commit”。

### 规则 5：禁止最后一次性提交

- **行为描述**：执行者先改完整个任务，最后一次性提交所有文件。
- **为什么是问题**：中间状态不可追踪，验证失败时需要回溯大量改动。
- **防止措施**：要求“每完成一个文件/模块/接口/维度就提交；如果连续多个改动未提交，必须先整理并拆分再继续”。

### 规则 6：禁止提交未验证改动

- **行为描述**：改完后不运行局部验证就提交。
- **为什么是问题**：提交历史中会混入语法错误、测试失败或不可运行状态。
- **防止措施**：每个 commit 前写明必须运行的验证命令；无法运行时提交信息和最终回复都要说明原因。

## 3. 以风险大为借口不改

### 规则 7：风险必须具体化

- **行为描述**：执行者说“风险大”，但不说明风险来自行为变化、接口变化、数据迁移还是验证缺失。
- **为什么是问题**：没有具体风险就无法判断该修改是否真的应停止。
- **防止措施**：要求把风险拆成“会改变什么行为、影响哪些调用方、缺少什么证据、需要用户决定什么”。

### 规则 8：优先做最小安全改动

- **行为描述**：看到大函数、重复逻辑或复杂依赖后完全不改。
- **为什么是问题**：很多目标可以通过局部命名、拆小函数、补测试、收窄异常处理等方式安全推进。
- **防止措施**：写明“能做局部、行为保持、可验证改动时必须继续；只有不可避免改变功能行为时才停下”。

### 规则 9：禁止把缺少完美验证当作不执行理由

- **行为描述**：因为没有全量测试或 CI 不完整，就拒绝做任何改动。
- **为什么是问题**：缺少完美验证不等于没有验证面，仍可使用语法检查、局部测试、静态检查和人工审计。
- **防止措施**：要求定义最低验证面；如果最终验证不足，必须说明已执行的替代验证和剩余风险。

## 4. 虚假完成

### 规则 10：禁止只给总结不列证据

- **行为描述**：最终回复说“已完成”，但没有列出改动文件、验证命令、commit 或未完成项。
- **为什么是问题**：用户无法确认任务是否真的完成，也无法定位结果。
- **防止措施**：最终回复必须包含完成项、验证结果、commit 列表、未完成项、跳过项和风险。

### 规则 11：禁止把计划当成果

- **行为描述**：输出计划、建议或待办，却暗示任务已经完成。
- **为什么是问题**：/goal 任务关注实际执行，不是只要方案。
- **防止措施**：要求区分“已完成改动”和“建议后续任务”；未落地内容必须列入未完成项。

### 规则 12：禁止忽略失败验证

- **行为描述**：验证命令失败后继续总结完成，或只说“有些测试失败”。
- **为什么是问题**：失败验证直接影响验收，必须被解释和处理。
- **防止措施**：要求记录失败命令、失败原因、已修复内容；如果不是本次改动导致，也要给出证据。

## 5. 过度保守

### 规则 13：禁止把简单改动升级成大型重构

- **行为描述**：为了“更稳妥”提出大改架构、换框架、重写模块，而不完成原目标。
- **为什么是问题**：过度设计会偏离用户边界，增加风险和成本。
- **防止措施**：写明“先完成用户目标的最小必要改动；大型重构只记录为后续建议”。

### 规则 14：禁止因边界不完美而停止全部任务

- **行为描述**：发现一个边界问题后停止整个任务，而不是继续处理明确范围。
- **为什么是问题**：局部不确定不应阻止已明确部分推进。
- **防止措施**：要求“明确范围继续执行；仅对不确定项停下或记录，不能影响其他可验证项”。

### 规则 15：禁止过度询问已可从代码验证的信息

- **行为描述**：本可通过阅读代码、运行测试确认，却反复向用户提问。
- **为什么是问题**：浪费用户注意力，也降低自主执行效率。
- **防止措施**：写明“能从代码、测试、文档或命令输出验证的信息必须自行验证；只有业务决策和不可推断预期才问用户”。

## 6. 范围漂移

### 规则 16：禁止顺手修范围外问题

- **行为描述**：执行者发现范围外坏味道或旧问题后直接修改。
- **为什么是问题**：范围漂移会引入未授权风险，也破坏 commit 主题。
- **防止措施**：要求范围外问题只记录路径、现象和建议，不进入本轮改动。

### 规则 17：禁止改动生成文件和第三方代码

- **行为描述**：为了让检查通过，直接修改生成物、锁文件或第三方拷贝代码。
- **为什么是问题**：这些文件通常不应手工改动，容易造成后续再生成冲突。
- **防止措施**：除非用户明确允许，否则排除生成文件、第三方目录和依赖锁文件；需要变更时先停下说明原因。

## 7. 验证绕过

### 规则 18：禁止删除或弱化测试

- **行为描述**：为了让测试通过，删除断言、跳过测试、降低覆盖目标或改期望值。
- **为什么是问题**：这会制造假绿灯，掩盖真实问题。
- **防止措施**：写明“不得删除、skip、xfail 或弱化现有测试；确需调整期望必须证明业务行为已按用户目标改变”。

### 规则 19：禁止忽略 lint/type 错误

- **行为描述**：质量命令失败后只说与本次无关，不给证据。
- **为什么是问题**：无法判断失败是否由本次改动引入。
- **防止措施**：要求保存验证输出摘要；若是既有失败，必须说明复现基线或给出与改动无关的证据。

### 规则 20：禁止用格式化掩盖功能改动

- **行为描述**：把格式化和逻辑改动混在一起，使审计者难以看出真正变化。
- **为什么是问题**：审计成本高，回滚时容易误伤。
- **防止措施**：格式化、重命名、逻辑改动分开提交；若工具自动格式化同一文件，必须在 commit message 中说明。

## 8. 生成强制要求的组合建议

- **代码质量优化/重构**：优先使用规则 1、2、4、5、8、16、20。
- **测试编写**：优先使用规则 3、4、6、10、18、19。
- **Bug 修复**：优先使用规则 6、7、10、12、14、19。
- **迁移/升级**：优先使用规则 1、3、4、7、12、16、17。
- **文档生成**：优先使用规则 3、10、11、16，并要求文档条目可追溯到文件或命令输出。

生成 `/goal` 指令时，强制要求通常保留 5-8 条即可。每条必须是可执行的禁止或必须规则，避免写成泛泛价值观。
