# /goal 指令 6 要素参考文档

本文件用于在生成 Codex CLI `/goal` 指令时深入检查需求完整度。任何最终指令都必须覆盖 Outcome、Verification Surface、Constraints、Boundaries、Iteration Policy、Blocked Stop Condition 六个要素；如果某个要素缺失或不可验证，必须一次性追问用户补充。

## 1. Outcome（目标结果）

### 定义

Outcome 是任务最终要交付的结果，必须具体到可以被检查、审计和验收。它回答“完成后项目中会多出什么、少掉什么、变好到什么程度、由什么证据证明”。

### 为什么重要

没有明确 Outcome，执行者容易把任务理解成开放式探索，做一些看起来有价值但不一定符合用户意图的改动。明确 Outcome 可以限制任务方向，也能让最终验证有依据。

### 好的示例

- 新建分支 `optimize/code-quality`，对 `src/`、`app/`、`services/` 下的非测试 Python 文件做 7 维度代码质量优化，并生成 `OPTIMIZE_REPORT.md`。
- 为 `src/services/payment.py` 和 `src/services/refund.py` 补齐 pytest 单元测试，新增不少于 20 个测试用例，并把关键异常路径覆盖到。
- 将项目内旧版配置读取逻辑迁移到新的 `Settings` 类，保留现有行为兼容，并更新相关调用点。

### 坏的示例

- 提高代码质量。
- 加一些测试。
- 修一下项目里的问题。
- 看看哪里能优化就优化。

### 检查清单

- 是否说明最终交付什么文件、功能、测试、报告或改动集合？
- 是否说明做到什么程度，例如数量、覆盖率、目录、模块或维度？
- 是否能在最终审计时判断“完成/未完成”？
- 是否包含合理分支名或允许生成器建议分支名？
- 是否避免了“尽量”“适当”“一些”这类不可验收表述？

## 2. Verification Surface（验证方式）

### 定义

Verification Surface 是确认每个改动正确、最终任务完成的证据集合。它通常包含命令、人工检查点、报告审计、编译检查、测试范围和最终验收步骤。

### 为什么重要

缺少验证面时，执行者容易只凭主观判断宣布完成。明确验证方式可以防止虚假完成，也能在出现失败时暴露真实阻塞点。

### 好的示例

- 每个 Python 文件改动后运行 `python -m py_compile <file>`；每组模块改动后运行对应单元测试；最终运行 `pytest -q`。
- 每新增一个接口后执行相关 API 测试，并用现有 lint 命令确认格式和类型检查通过。
- 每完成一个迁移切片后运行旧用例和新用例，最终检查迁移报告覆盖所有调用点。

### 坏的示例

- 确保没问题。
- 自己检查一下。
- 跑测试。
- 通过验证即可。

### 检查清单

- 是否给出具体命令、文件检查、报告审计或其他证据？
- 是否包含每次改动后的局部验证？
- 是否包含全部完成后的最终验证？
- 是否说明验证失败时要修复、记录还是停下？
- 是否避免只写“测试通过”但不说明运行什么？

## 3. Constraints（约束）

### 定义

Constraints 是任务执行中必须遵守的限制，说明不能改什么、不能破坏什么、必须保持什么兼容性和风格。

### 为什么重要

约束可以阻止执行者为了达成目标而做过度改动，例如改业务逻辑、删除测试、引入不必要依赖或重写架构。好的约束能让改动保持最小必要范围。

### 好的示例

- 不修改测试代码以外的业务逻辑，不引入新依赖，不改变公开 API。
- 不做大重构，不改数据库结构，不修改自动生成文件。
- 保持现有错误码、接口响应结构和日志字段兼容。

### 坏的示例

- 小心一点。
- 不要乱改。
- 尽量保持稳定。
- 看情况处理。

### 检查清单

- 是否明确禁止修改的文件、模块、依赖或行为？
- 是否说明兼容性要求，例如 API、数据格式、错误码、配置项？
- 是否限制重构规模和新依赖？
- 是否说明不能为了通过验证弱化测试或校验？
- 是否足以阻止“顺手扩大改动”？

## 4. Boundaries（边界）

### 定义

Boundaries 是任务的作用范围，明确哪些目录、文件、模块、语言、测试集或文档在范围内，哪些内容排除在外。

### 为什么重要

没有边界时，执行者可能在项目中无限扩展，既浪费时间，也增加风险。边界能帮助执行者列出候选文件，并判断发现的新问题是否属于本次任务。

### 好的示例

- 仅处理 `src/`、`app/`、`services/` 下的 `.py` 文件，排除 `tests/`、迁移脚本、生成文件和第三方代码。
- 只为 `src/services/` 的业务逻辑补测试，对应测试文件放在 `tests/services/`。
- 仅迁移配置读取相关调用点，不处理日志、监控和部署脚本。

### 坏的示例

- 整个项目都看看。
- 有问题就改。
- 相关的都处理。
- 你自己判断范围。

### 检查清单

- 是否能据此列出候选文件或模块？
- 是否说明包含范围和排除范围？
- 是否说明遇到范围外问题时只记录还是另行询问？
- 是否避免“相关”“附近”“顺便”这类模糊边界？
- 是否与 Outcome 的目标范围一致？

## 5. Iteration Policy（迭代策略）

### 定义

Iteration Policy 是执行任务的步进策略，说明每一步改动的粒度、每步后的验证动作、提交动作和预期 commit 数量。

### 为什么重要

没有迭代策略时，执行者容易一次性做大量改动，难以回滚和审计；也容易把多个独立目标塞进一个 commit。明确迭代策略能让长期任务稳定推进。

### 好的示例

- 每完成一个独立改动立即运行对应验证，验证通过后 `git add` + `git commit`，再继续下一个。
- 每个模块作为一个切片，先补测试再改实现，每个切片单独 commit，预期 6-10 个 commit。
- 每迁移一个调用点就运行相关测试并提交，最终再做一次全局搜索确认无遗漏。

### 坏的示例

- 做完后提交。
- 最后一起 commit。
- 按你习惯来。
- 一口气改完。

### 检查清单

- 是否说明每步处理一个文件、模块、接口、维度还是问题？
- 是否要求每步后执行验证？
- 是否要求每个独立改动单独 commit？
- 是否给出预期 commit 数量范围或偏离时解释规则？
- 是否说明最终合并和推送动作？

## 6. Blocked Stop Condition（受阻停止条件）

### 定义

Blocked Stop Condition 是执行者在遇到不确定性、风险或缺失信息时的处理规则，明确什么时候可以跳过、什么时候必须停下问人、跳过项上限是多少。

### 为什么重要

执行者常见失败方式是遇到困难就跳过，或者在缺少业务决策时凭空猜测。受阻停止条件既限制偷懒，也防止越权决策。

### 好的示例

- 只有修改后必然导致功能行为变化时才允许跳过；跳过项必须记录证据，未修改项不超过 5%。
- 如果无法从现有代码或测试推断业务预期，必须停下说明缺口并等待用户确认。
- 第三方依赖版本冲突、数据库结构变更、公共 API 变更必须停下，不得自行决定。

### 坏的示例

- 有问题就跳过。
- 风险大就别改。
- 不确定你自己判断。
- 遇到困难再说。

### 检查清单

- 是否定义允许跳过的严格条件？
- 是否定义必须停下问人的条件？
- 是否要求记录证据、原因和影响范围？
- 是否给出跳过项上限或“不能跳过”的规则？
- 是否防止执行者以笼统风险为借口不做范围内工作？

## 7. 整体检查清单

最终生成 `/goal` 指令前，逐项确认：

- 6 个要素都已覆盖，且不是泛泛而谈。
- 任务概述能在一段话里讲清目标、验证、约束、边界、迭代和受阻规则。
- 强制要求能堵住跳过困难问题、合并 commit、虚假完成、过度保守等行为。
- 提交规则包含 commit 粒度、message 格式、示例、合并和推送指令。
- 报告或文档类任务包含输出文件格式和表格结构。
- 多维度任务包含每个维度的处理规则或至少说明如何逐项覆盖。
- 纯文本输出可直接复制粘贴，不使用 Markdown 代码块包裹。
